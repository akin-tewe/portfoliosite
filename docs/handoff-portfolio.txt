### Rewrite Context

Sifty case study was COMPLETELY REWRITTEN (tasks 173-175) based on the Meta
feedback. The original was a feature showcase ("here's what I built"). The new
version is a problem-framed narrative following the Found& pattern.

### Sections
1. Hero + metadata (Role: Sole Designer & Developer, Timeline: 2 Months,
   Type: Web Application, Status: Live at sifty.app)
2. Overview
3. THE PROBLEM — three named breakdowns: invisible boundaries, tools that
   manage tasks not scope, documentation friction
4. RESEARCH — competitive analysis of Trello/Asana/Notion/Basecamp through
   scope visibility lens. Trello dashboard screenshot. Stats from MIT Sloan
   and PwC (25-40% budget overrun, 42% insufficient requirements, 23% clear
   acceptance criteria). Research questions that shaped design.
5. THE OPPORTUNITY — scope as primary interface, not one feature among many
6. DEFINING SCOPE — Included vs. Deliberately Excluded side-by-side with
   rationale baked into each bullet. This section is one of the strongest.
7. SOLUTION DESIGN — 5 numbered subsections, each with tension → decision →
   tradeoff pattern:
   1. Timeline as primary interface (vertical vs horizontal Gantt tradeoff)
   2. Making exclusions visible (structured vs unstructured notes tradeoff)
   3. Client view: read-only by design (removed vs disabled controls)
   4. Form design: validation on submit (on-blur vs on-change vs on-submit)
   5. Quick-add: 30-second documentation (speed vs completeness tradeoff)
8. RESPONSIVE DESIGN — mobile not compressed desktop, separate experience.
   Timeline stem at 65% scale, cards full-size. Complex editing desktop-only.
9. BUILT WITH — tech stack pills
10. Closing — reflection paragraph about investing more in user testing

### Copy Guidelines
- No em dashes anywhere (tasks 175, 185)
- First person, direct voice, short sentences
- No AI writing patterns: no triple-parallel lists, "leveraging",
  "Furthermore", "In conclusion". Voice pass (task 177) ensured natural tone.
- TradeoffBlock component for explicit design decision rationale
